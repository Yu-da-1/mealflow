# /ui

Pencil MCP で開いているデザインを読み取り、実装タスクを `tasks/current.md` に書き起こす。

---

## 1. デザインを確認する

まず `mcp__pencil__get_editor_state` でエディタの状態を確認する。

- ファイルが開いていない場合 → リポジトリ内の `**/*.pen` を Glob で検索し、見つかったファイルを `mcp__pencil__open_document` で開く。複数ある場合はファイル名をユーザーに提示して選んでもらう
- 開いている場合 → そのまま次へ

次に以下を順番に実行してデザインの全体像を把握する。

1. `mcp__pencil__get_screenshot` でスクリーンショットを取得して画面構成を視覚的に把握する
2. `mcp__pencil__batch_get` でルートノードを取得してコンポーネント構成を確認する（必要に応じて深掘りする）

---

## 2. デザインを読み解く

取得した情報をもとに以下を整理する。

- **画面名・機能名**: どの画面・機能のデザインか
- **UIコンポーネント一覧**: 画面に存在するコンポーネント（ボタン、フォーム、リスト等）
- **インタラクション**: タップ・入力・遷移などのユーザー操作
- **データ表示**: 何のデータを表示・入力しているか
- **既存コードとの対応**: `src/features/` 配下の既存コンポーネントと照合し、新規作成か修正かを判断する

不明点があればユーザーに確認してから次へ進む。

---

## 3. `tasks/current.md` に書き起こす

以下のフォーマットに従って `tasks/current.md` を生成・上書きする。

```markdown
## [フェーズ番号]: [機能名]

- ブランチ: `feature/[kebab-case-name]`
- PR: このグループ完了後に1PR

### 完了条件

- （デザインから読み取った「この機能が完成した状態」を箇条書き）

### タスク

- [ ] 型定義: （追加・変更が必要な型）
- [ ] domain層: （純粋関数があれば）
- [ ] repository層: （DBアクセスが必要であれば）
- [ ] APIルート: （エンドポイントが必要であれば）
- [ ] UIコンポーネント: （新規・修正するコンポーネント名）
- [ ] hooks: （必要なカスタムhook）
- [ ] 手動確認: （実機・ブラウザでの動作確認）
```

**ルール:**

- `docs/design.md` の該当セクションがあれば参照して完了条件に反映する
- MVP範囲外の機能は含めない（`docs/design.md` セクション2を確認する）
- レイヤー順（型 → domain → repository → API → UI）でタスクを並べる
- 実装の判断に迷う点は `tasks/current.md` の末尾に `### 確認事項` として書き出す

---

## 4. ユーザーに確認する

書き起こした内容をユーザーに提示し、「このタスクで進めてよいか」を確認する。
修正があれば反映してから終了する。

確認が取れたら `/implement` に進む準備完了であることを伝える。
