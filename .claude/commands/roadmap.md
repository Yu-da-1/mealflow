# /roadmap

長期的な方向性を議論し、`docs/roadmap.md` に記録する。
**実装・backlogへの追記は行わない。** このコマンドは長期計画の議論と記録のためだけに使う。
backlogへの移動は `/plan-next` で行う。

必ず 0 → 1 → 2 → 3 → 4 の順で進めること。

---

## 0. ユーザーインプット（任意）

気になっていること・問題点・やりたいことがあれば確認する。

> 「追加したい機能や気になっている問題点はありますか？なければスキップします。」

ユーザーからインプットがあれば、以降のステップで背景として考慮する。

---

## 1. 現状把握

以下を読んで現状を整理する。

- `tasks/backlog.md`: 完了済み・未着手のPhaseを把握する
- `docs/design.md`: MVPスコープと将来対応を確認する
- `docs/competitive-analysis/index.md`: 存在する場合、競合分析の示唆を参照する（なければスキップ）
- 必要に応じて `src/` を探索し、実装済みの範囲を確認する

整理した内容をユーザーに見せる。

```
### 現状サマリー
- 完了済み: Phase X, Y, Z
- バックログ残: Phase A（未着手）, Phase B（未着手）
- MVPとして未実装: ○○, ××
```

---

## 2. 提案

現状サマリーとユーザーインプットをもとに、次に着手すべきPhaseを提案する。

提案には必ず以下を含める。

- **優先順位と理由**: なぜこの順番か（ユーザー価値・技術的依存・リスク など）
- **複数の方向性がある場合**: 選択肢として提示し、どれを選ぶかはユーザーに委ねる

```
### 提案

**方向性A: ○○を先に進める**
理由: 〜〜〜

**方向性B: ××を先に進める**
理由: 〜〜〜

どちらを優先しますか？
```

---

## 3. フィードバック・確定

ユーザーのフィードバックをもとに内容を調整し、合意を取る。

- 優先順位の変更
- スコープの拡縮（やること・やらないことの調整）
- Phase・タスクの粒度の調整

合意が取れたら次のステップへ進む。

---

## 4. docs/roadmap.md に記録

合意した内容を `docs/roadmap.md` に書き出す（または更新する）。
実装ブランチに関係なく編集できる長期計画ドキュメントとして管理する。

```markdown
# ロードマップ

最終更新: YYYY-MM-DD

## やりたいこと（優先度順）

（方向性・将来機能のリスト。タスクに分解できるレベルになったら /plan-next でbacklogに移す）

## 検討中・保留

（やりたいが優先度・難易度で保留しているもの）
```

backlogへの追記は行わない。`/plan-next` で行う。
