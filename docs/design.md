# 冷蔵庫管理・献立提案アプリ MVP 設計仕様書

## 1. 概要

本アプリは、冷蔵庫内の在庫を管理し、食品ロスを減らし、日々の献立決定を簡単にすることを目的とする。

### 目的
- 冷蔵庫の中身を可視化する
- 賞味期限・消費期限切れによる廃棄を減らす
- 毎日の献立決定コストを下げる

### 想定ユーザー
- 忙しい中でも日常的に料理をする人
- 何が冷蔵庫に入っているか把握しづらい・期限管理が面倒・献立決定が負担な人

---

## 2. MVP のスコープ

### 含む機能
- 在庫の登録・更新・削除（CRUD）
- 賞味期限 / 消費期限の管理
- 期限が近い食品のアプリ内表示
- レシピ提案（3件）
- 同じレシピが続けて提案されない制御
- 食品単位の集約表示とロット単位の詳細表示

### 含まない機能（将来対応）
- 健康状態に基づく推薦 / OCR・バーコード読み取り
- 購入履歴との自動連携 / Push通知 / 買い物リスト / 家族共有

---

## 3. コアユーザーフロー

1. 食品を購入する
2. アプリに食品を登録する
3. 期限が自動入力される（必要なら編集可能）
4. 在庫・期限を確認する
5. アプリがレシピを3件提案する
6. レシピを選択する
7. 使用食材を確認する
8. 在庫が更新される

---

## 4. 画面構成

### 4.1 ホーム
- 期限が近い食品（当日 / 翌日）
- 在庫サマリ
- おすすめレシピ3件
- 食品追加ボタン

表示優先度: 期限リスク → 在庫 → レシピ提案

### 4.2 食品一覧
- 食品ごとに集約表示（例: 卵 16個）
- アコーディオン展開でロット一覧（数量・期限・設定元）
- ロット単位で編集・削除

### 4.3 食品登録
- 食品名（検索 + オートコンプリート）・数量・購入日・期限日
- 期限は食品マスタから自動設定。ユーザーが上書き可能
- 期限種別（賞味/消費）と設定元（自動/手動）を表示

### 4.4 レシピ画面
- おすすめレシピ3件
- 選択後: 使用食材のチェックリスト表示 → 確定で在庫を減算

## 5. ドメインロジック

### 5.1 期限自動設定

- `expiry_date` 未入力時 → `food_masters.default_expiry_days + purchased_at` で計算
- 自動設定 → `expiry_source = 'estimated'`
- ユーザー入力 → `expiry_source = 'manual'`
- UIでは賞味期限・消費期限をまとめて「期限」として表示する

デフォルト期限の例:
| 食品 | 日数 |
|---|---|
| 卵 | 14日 |
| 牛乳 | 7日 |
| 豆腐 | 4日 |
| 鶏肉 | 2日 |
| 豚肉 | 3日 |
| ほうれん草 | 3日 |
| もやし | 2日 |

### 5.2 レシピ一致判定

- 必須食材（`is_required = true`）がすべて在庫に存在するレシピが対象
- 調味料は判定対象から除外する
- 数量の厳密一致は見ない
- `active` な在庫のみ対象

一致ルール:
1. `recipe_match_key` の完全一致を優先
2. `parent_recipe_match_key` の上位カテゴリ一致も許可

例: `chicken_thigh` は `chicken_thigh` にも `chicken` にも一致できる

### 5.3 レシピスコアリング

```
スコア = (期限が当日・翌日の食材数 × 20) + (一致する在庫食材数 × 10)
```

- 直近3日で提案済み（`recipe_recommendation_logs`）のレシピは除外
- 上位3件を返す

### 5.4 在庫減算

1. ユーザーがレシピを選択・食材を確認して確定する
2. 同一食品の複数ロットは `expiry_date` が近い順に優先減算
3. `expiry_date` が同一または未設定の場合は `purchased_at` が古い順
4. 数量が0になったロットは `status = 'consumed'` に更新

## 6. 画面別詳細仕様

### 6.1 ホーム画面

#### 期限が近い食品セクション
- `active` な在庫のみ対象
- 当日分を最優先（赤系）、翌日分を次（黄系）
- 最大5件。超える場合は「もっと見る」で食品一覧へ

#### 在庫サマリ
- `active` 在庫の食品種類数を表示（例: `12 種類の食品があります`）

#### おすすめレシピセクション
- 上位3件。各カードにレシピ名・提案理由・調理時間を表示
- 提案理由例: `期限が近い卵を使えます` / `今ある食材だけで作れます`

#### 状態
- loading: スケルトン表示
- empty: 在庫0件 → `食品を追加するとレシピ提案が表示されます`
- error: 再読み込み導線を表示

### 6.2 食品一覧画面

#### 一覧表示
- 集約キー: `food_master_id`
- 表示項目: 食品名・合計数量・最も近い期限

#### 並び順
1. 期限当日 → 2. 期限翌日 → 3. 期限が近い順 → 4. 期限なし or 遠い順

#### アコーディオン詳細（各ロット）
- 数量・購入日・期限日・期限種別・設定元（自動/手動）・編集・削除ボタン

#### 状態
- loading: 一覧スケルトン
- empty: `まだ食品が登録されていません`
- error: `食品一覧の取得に失敗しました`

### 6.3 食品登録画面

#### バリデーション
- 食品名: 必須
- 数量: 必須、1以上の整数
- 購入日: 必須（初期値は当日）
- 期限日: 任意

#### 保存時の挙動
- 期限日未入力 → `food_masters.default_expiry_days` で自動設定（`expiry_source = estimated`）
- ユーザー入力 → `expiry_source = manual`
- `default_expiry_type` を `inventory_lots.expiry_type` にコピー

#### 状態
- saving: 保存中ボタン状態
- success: 一覧またはホームへ戻す
- error: フォーム下部にエラー表示

### 6.4 レシピ画面

#### 一覧表示
- 3件固定。各カードにタイトル・提案理由・調理時間・詳細ボタン

#### レシピ選択後
- `recipe_ingredients` をもとに使用食材をチェックリスト表示
- 初期状態はすべてチェック済み。ユーザーが外せる
- 確定時、チェック済み食材のみ在庫減算対象

#### 状態
- loading: レシピカードスケルトン
- empty: `現在の在庫で作れるレシピがありません`
- error: `レシピの取得に失敗しました`

## 7. フロントエンド実装メモ

### 状態管理
- データ取得・更新: TanStack Query
- フォーム: React Hook Form + Zod

### コンポーネント分割
- `HomeExpirationSection` / `HomeRecipeSection`
- `InventoryList` / `InventoryAccordionItem`
- `FoodForm`
- `RecipeCard` / `RecipeConsumeDialog`

### 型の方針
DB由来の型とUI用の型を分ける。

| 型名 | 用途 |
|---|---|
| `FoodMasterRow` | DBそのまま |
| `InventoryLotRow` | DBそのまま |
| `InventorySummaryItem` | 集約済みUI用 |
| `RecommendedRecipe` | レシピ提案UI用 |

---

## 8. 残課題

- 食品マスタ seedデータ（100〜150件）: category / recipe_match_key / default_expiry_days の定義
- レシピ seedデータ（30〜50件）: recipe_ingredients との整合が必要
- 推薦ロジックの詳細化: 同カテゴリ料理の連続提案を避けるか検討
- 通知拡張: 将来的に Web Push または Discord 通知へ対応可能な設計にする

## 10. アーキテクチャ方針

詳細は CLAUDE.md を参照。ここでは背景と目的のみ記載する。

### 方針の背景
- 将来的な Expo / React Native 対応を前提とする
- Web と Mobile は UI のみ分離し、API・ドメインロジック・型は共通化する

### 設計原則
1. UIはDBを直接操作しない
2. APIはロジックを持ちすぎない
3. ドメインロジックは純粋関数として実装する
4. DBアクセスはrepository層に閉じる
5. HTTP APIを唯一のインターフェースとする
