# 004 parent_recipe_match_key の使い分け方針を決める

**日付**: 2026-04-02
**ステータス**: Accepted

## 決定内容

既存レシピの `recipe_ingredients` はそのまま維持する。新規レシピは「どの部位でも作れる料理」にのみ親キー（例: `chicken`）を使い、部位が重要な料理は引き続き具体キー（例: `chicken_thigh`）を使う。

## 背景

食品マスタに `parent_recipe_match_key` を設定することで、`chicken_thigh` を持っていれば `chicken` キーを要求するレシピにもマッチするようになる。
しかし「`chicken` キーをレシピに使う」かどうかは、その料理が部位を問わないかどうかで変わる。
レシピ side の `recipe_match_key` をどう書くか、既存レシピをどう扱うかの方針を決める必要があった。

## 検討した選択肢

| 選択肢                                 | 概要                                      | 問題点                                                                                       |
| -------------------------------------- | ----------------------------------------- | -------------------------------------------------------------------------------------------- |
| 既存レシピを親キーに書き換える         | `chicken_thigh` → `chicken` に一括変換    | 部位にこだわるレシピ（親子丼・から揚げ）でも「鶏むね肉」でマッチしてしまい、仕上がりが変わる |
| 既存レシピに親キー行を追加する         | `chicken_thigh` と `chicken` の両行を持つ | 重複・冗長。どちらが必須か曖昧になる                                                         |
| **既存はそのまま・新規のみ方針を適用** | 既存は変更せず、新規レシピで部位を選ぶ    | ルールが明快でデータの意図が保たれる                                                         |

## 採用理由

1. **既存レシピの意図を保つ**: 親子丼・から揚げはもも肉が適切であり、むね肉でも「一致」となるのは不自然。
2. **ルールが明快**: 「部位が重要 → 具体キー」「どちらでも可 → 親キー」という判断基準が開発者・AIどちらにも理解しやすい。
3. **変更コストがゼロ**: 既存 SQL を触らないので回帰リスクがない。

## 残課題・制約

- 判断基準（部位が重要かどうか）はレシピ追加者の裁量に委ねる。迷ったら具体キーを使う。
