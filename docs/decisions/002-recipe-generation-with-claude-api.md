# 002 レシピ生成に Claude API を使用する

## 決定内容

外部レシピサービスからの取得ではなく、Claude API（Haiku モデル）を使って在庫食材からレシピをリアルタイム生成する方式を採用する。

## 背景・論点

### 検討した選択肢

| 選択肢                  | 概要                               | 問題点                                                                                                                   |
| ----------------------- | ---------------------------------- | ------------------------------------------------------------------------------------------------------------------------ |
| 楽天レシピ API          | 無料・日本語・手順付きレシピあり   | カテゴリ検索中心で「在庫食材から検索」が苦手。食材名の正規化（"鶏もも肉200g" → `chicken_thigh`）が必要で実装コストが高い |
| クックパッド / クラシル | 本物のレシピが豊富                 | 公開 API なし。スクレイピングは利用規約違反                                                                              |
| Spoonacular             | API あり                           | 英語中心で和食が弱い                                                                                                     |
| seed データ手動作成     | シンプル                           | 量・品質ともに限界がある                                                                                                 |
| **Claude API（採用）**  | 在庫食材を渡してその場でレシピ生成 | API コストあり（下記参照）                                                                                               |

### コスト試算

- 1回あたりの入力: 約 400 トークン（システムプロンプト + 在庫リスト）
- 1回あたりの出力: 約 1,500〜2,000 トークン（レシピ3件 + 手順）
- Haiku モデル使用時: **約 1円/回**
- 個人利用（1日1回）: **約 30円/月**

## 採用理由

1. **food_master との整合性が自然に取れる** — `recipe_match_key` をそのまま渡せるため、食材名の正規化処理が不要
2. **在庫ベースの提案と相性がよい** — 「今ある食材」に特化したレシピをその場で生成できる
3. **コストが現実的** — 個人用途では月数十円レベル
4. **スキーマ変更が不要** — 既存の `recipes.instructions` カラムにそのまま入る

## 実装方針

- モデル: `claude-haiku-4-5-20251001`（コスト優先）
- タイミング: `GET /api/recipes/recommended` 呼び出し時に生成
- キャッシュ: 在庫の変化がなければ前回の生成結果を再利用（API 呼び出しを抑制）
- フォールバック: API エラー時は既存の seed レシピを返す

## 残課題

- キャッシュ戦略の詳細設計（在庫変化の検知方法）
- プロンプト設計（日本語レシピ・手順の品質確保）
- API キーの管理（環境変数 `ANTHROPIC_API_KEY`）
